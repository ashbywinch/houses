SYSTEM ARCHITECTURE & ENGINEERING BRIEF:
Browser-to-Spreadsheet Ingestion & Enrichment Engine
1. Objective
You are being tasked with building a lightweight, local automation broker that allows a user browsing property listings on Firefox (Ubuntu) to extract property details with one click, run deterministic transit and school calculations, and automatically inject the structured data into a collaborative Google Sheet.
You do not yet have read/write access to our active spreadsheet workbook, we'll need to figure out how to set that up. 
2. The Core Human Needs & Context
This tool is being built to help three people (Simon, Lorena, and the user) collaborate on finding a family house. They are actively debating properties on WhatsApp, but need a single "source of truth" database to make decisions.
* ** Simon's Work Anchor:** Pimlico / Victoria Area (Postcode: SW1V 2QQ / 171 Victoria Street). Commutes via public transit.
* ** Lorena's Work Anchor:** Aldgate / City of London (Postcode: EC3A 7LP). Commutes via public transit.
* ** Bracknell Office Anchor:** Postcode RG12 8YA. Commutes via personal car (reasonably modern petrol hatchback averaging 45 mpg at £1.45/L).
* ** Boy's Schooling Constraint:** The family has a boy. Therefore, all extracted primary and secondary schools must accept boys. If a local catchment area's closest high school is girls-only (e.g., Wycombe High or Didcot Girls'), the system must automatically identify and substitute the closest co-educational or boys-only alternative (e.g., John Hampden Grammar or St Birinus School) and calculate walk/bus times for that substitute instead. All schools must be non fee paying.
3. Critical Constraints Discovered
A. The Scraping Roadblock
You cannot scrape Rightmove links directly using raw urllib, requests, or programmatic headless scrapers. Rightmove utilizes Human Security (formerly PerimeterX), which blocks automated AI cloud requests instantly.
* The Workaround: The user will view the property pages natively in Firefox on Ubuntu. They will use a free, open-source browser sidepanel (Page Assist) configured as a Bring Your Own Key (BYOK) assistant using their own OpenRouter/LLM keys. Page Assist will extract the raw HTML/text of the active tab locally, bypassing the scraper blocks.
B. The UX Preservation Constraint (Why we can't write directly to the main sheet)
Google Sheets is highly collaborative, but terrible for running raw chat logs. If our automation script writes directly to the user's active rows:
1. It risks over-writing custom conditional formatting, cell fills, or custom dropdowns.
2. It will clobber manual comments or WhatsApp logs pasted by Simon, Lorena, or the user.
3. It creates data collision if two people edit a row at the same time the script triggers.
4. The Solution: Split-Sheet "Database" Architecture
To keep bot edits and human edits completely separated, you must structure the Google Sheet workbook into two distinct tabs:
1. Tab 1: Properties (Human) * This is the user-facing dashboard. Humans manually enter the Property Name, the Rightmove Link, the Status, and their Comments.
   * All commute times, travel costs, and school distances are populated using native, live Google Sheets formulas: =IFERROR(XLOOKUP($B2, 'AI_Data_Source (Bot)'!$A:$A, 'AI_Data_Source (Bot)'!$E:$E), "")
   * This ensures the human sheet stays responsive, lets users color-code cells red/green, and guarantees that your automation script never writes to this tab.
2. Tab 2: AI_Data_Source (Bot)
   * This is a flat, read-only data warehouse. Your local python server has exclusive write-access to this tab.
   * Its unique primary key is the Rightmove Link in Column A.
5. Suggested Technical Approach: The Webhook Broker
You need to write a lightweight, local API endpoint (using FastAPI or Flask) running on the user's local Ubuntu machine.
The Processing Loop:
1. The Trigger: The user opens a Rightmove page, opens the Page Assist sidepanel, and runs a custom System Prompt.
2. The Extraction: The browser AI reads the page content and extracts a clean, minimal JSON object containing:
   * url (The active Rightmove link)
   * postcode (The property postcode)
   * bedrooms (Number of beds)
   * price (The listing price)
3. The API Call: The browser AI executes a local HTTP POST request (via a curl command or web hook) sending this JSON to your local running python server (e.g., http://127.0.0.1:8080/inject-property).
4. The Local Enrichment: Your python script receives the payload and:
   * Calculates the transit commute times to SW1V 2QQ and EC3A 7LP using transit baselines.
   * Calculates the petrol fuel cost for the round trip to the Bracknell office (RG12 8YA).
   * Queries local school registries (or uses a targeted API call) to identify the closest boy-eligible primary and secondary schools.
   * Identifies public transport/bus route profiles if school walking times exceed 20 minutes.
5. The Database Write: Your script uses gspread to write this newly enriched row directly to the bottom of the AI_Data_Source (Bot) tab. The live Excel/Google Sheets formulas in the Human tab will instantly update across everyone's devices.
6. Your Immediate Actions
Please inspect the active spreadsheet. Then:
1. Write the complete, production-ready Python FastAPI script (injector_server.py) to run locally on Ubuntu.
2. Provide the exact, highly engineered System Prompt we should save in Page Assist to ensure it parses the webpage text accurately and outputs a structured curl command to feed your server.
3. Implement robust error handling and exponential backoffs for all outbound API/LLM calls.
